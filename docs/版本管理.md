## 版本号规范

采用语义化版本控制（Semantic Versioning 2.0.0）：

```
主版本号.次版本号.修订号.构建号
MAJOR.MINOR.PATCH.BUILD
```

### 版本号说明
- **主版本号 (MAJOR)**: 当你做了不兼容的 API 修改
- **次版本号 (MINOR)**: 当你做了向下兼容的功能性新增
- **修订号 (PATCH)**: 当你做了向下兼容的问题修正
- **构建号 (BUILD)**: 自动构建编号

### 示例
- `1.0.0.0` - 初始版本
- `1.2.0.0` - 当前版本，新增了子网计算等功能
- `1.2.1.0` - 修复bug版本
- `2.0.0.0` - 重大更新版本

## 版本发布流程

### 1. 开发阶段
- 在开发分支进行功能开发
- 提交代码时使用清晰的提交信息

### 2. 版本准备
1. 更新版本号
   - 修改 `src/NETKit/NETKit.csproj` 中的版本信息
   - 确保 `AssemblyVersion`、`FileVersion` 和 `Version` 一致

2. 更新文档
   - 更新 `README.md`
   - 更新版本历史记录

### 3. 构建和测试
```bash
# 使用构建脚本进行发布
build-release.bat
```

### 4. 版本标记
```bash
# 创建版本标签
git tag -a v1.2.0 -m "Release version 1.2.0"
git push origin v1.2.0
```

### 5. 发布
- 将构建好的可执行文件上传到发布位置
- 更新发布说明

## 分支管理策略

### 主要分支
- `master`: 主分支，包含稳定的发布版本
- `develop`: 开发分支，包含最新的开发功能
- `feature/*`: 功能分支，用于开发新功能
- `hotfix/*`: 热修复分支，用于紧急修复

### 分支命名规范
- 功能分支: `feature/功能名称`
- 修复分支: `hotfix/修复内容`
- 发布分支: `release/版本号`

## 提交信息规范

使用以下格式编写提交信息：

```
类型(范围): 简短描述

详细描述（可选）

相关问题编号（可选）
```

### 提交类型
- `feat`: 新功能
- `fix`: 修复bug
- `docs`: 文档更新
- `style`: 代码格式调整
- `refactor`: 代码重构
- `test`: 测试相关
- `chore`: 构建过程或辅助工具的变动

### 示例
```
feat(subnet): 添加子网计算功能

- 实现IPv4子网计算
- 支持CIDR和子网掩码转换
- 添加VLSM计算支持

Closes #123
```

## 注意事项

1. **版本一致性**: 确保所有地方的版本号保持一致
2. **向下兼容**: 次版本号更新时要保持向下兼容
3. **测试覆盖**: 每次发布前进行充分测试
4. **文档同步**: 版本更新时同步更新相关文档
5. **备份策略**: 重要版本要做好备份
